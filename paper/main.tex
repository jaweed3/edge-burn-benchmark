\begin{abstract}
Edge inference on ARM-based single-board computers is critical for deploying deep learning in resource-constrained environments. This paper presents a comparative steady-state warm-cache latency benchmark of three inference frameworks---Burn (via tract-onnx), TensorFlow Lite (TFLite), and ONNX Runtime (ORT)---on a Raspberry Pi 5 (Cortex-A76) using MobileNetV2. We evaluate single-threaded and multi-threaded latency, throughput, memory usage, CPU utilization, and thermal characteristics under a controlled 2000-iteration protocol with IQR-based outlier removal and 95\% bootstrapped confidence intervals. Results show TFLite achieves the lowest latency (22.2 ms single-threaded, 10.7 ms with 4 threads), while Burn demonstrates exceptional memory efficiency (41.8 MB peak RSS). ONNX Runtime offers a balanced trade-off with 45.4 ms single-threaded latency and 100 MB memory footprint. All frameworks produce valid finite outputs on the evaluated model. The findings establish tract-onnx as a viable Rust-native ONNX runtime for ARM64 edge deployment where memory constraints are paramount, and provide reproducible baselines for framework selection.
\end{abstract}

\section{Introduction}
\label{sec:intro}
%----------------------------------------------------------------------

The proliferation of deep learning on edge devices has created a growing demand for efficient inference runtimes that balance latency, memory, and power consumption~\cite{howard2017mobilenets, sandler2018mobilenetv2}. Single-board computers such as the Raspberry Pi series have become popular platforms for edge AI prototyping due to their affordability and accessibility. However, selecting an appropriate inference framework for these resource-constrained platforms requires rigorous comparative benchmarking under controlled conditions.

Prior work includes Ignatov et al.~\cite{ignatov2019ai}, who benchmarked mobile AI accelerators across multiple frameworks, and AskariHemmat et al.~\cite{askarihemmat2020systematic}, who evaluated TFLite and ONNX Runtime on edge devices. Almeida et al.~\cite{almeida2023benchmarking} compared inference engines for embedded deployment, while Patil et al.~\cite{patil2021comprehensive} surveyed edge inference optimization techniques. More recently, the Burn framework~\cite{burn2024} introduced Rust-native deep learning with ONNX import capabilities, but its inference performance on ARM64 hardware has not been systematically benchmarked.

Three methodological considerations distinguish this work from prior benchmarks. First, we explicitly measure steady-state warm-cache latency after a 1000-iteration warmup---single-tensor repetition eliminates input-variance confounding, but we frame results accordingly as cache-warm steady-state performance rather than general-purpose throughput. Second, we simultaneously collect latency, memory, CPU utilization, and temperature under a single controlled protocol. Third, we provide a fully reproducible pipeline with code, data, and figures publicly available.

\section{Methods}
\label{sec:methods}

\subsection{Hardware Platform}

Experiments were conducted on a Raspberry Pi 5 running Debian GNU/Linux (kernel 6.18.34+rpt-rpi-2712) with the following specifications:
\begin{itemize}
    \item \textbf{SoC:} Broadcom BCM2712, quad-core ARM Cortex-A76 at 2.4 GHz
    \item \textbf{Memory:} 8 GB LPDDR4X
    \item \textbf{Storage:} 64 GB microSD (UHS-I)
    \item \textbf{Cooling:} Passive heatsink, no active fan
\end{itemize}

\subsection{Framework Configuration}

\noindent\textbf{Burn (tract-onnx):} Burn~\cite{burn2024} is a Rust-native deep learning framework. We used tract-onnx (v0.21), a pure-Rust ONNX inference engine. The \texttt{mobilenetv2-7.onnx} model was loaded at runtime via \texttt{tract\_onnx::onnx()} and optimized through tract's constant-folding and shape-inference passes. Inference uses f32 precision. tract-onnx uses a single-threaded execution plan regardless of thread configuration.

\noindent\textbf{TensorFlow Lite:} TFLite was used via the Python \texttt{tensorflow} package (v2.\textit{x}) with the XNNPACK delegate. Interpreter threads were set via \texttt{setNumThreads()}. Input preprocessing: resize to $224\times224$, pixel values in $[0,1]$.

\noindent\textbf{ONNX Runtime:} ORT was used via \texttt{onnxruntime} (CPU execution provider) with thread count via \texttt{SessionOptions.intra\_op\_num\_threads}. Input preprocessing: resize to $224\times224$, normalize to $[-1,1]$ via $(x/127.5)-1$, NCHW layout.

\subsection{Model and Input}

All frameworks ran MobileNetV2~\cite{sandler2018mobilenetv2} at $224\times224\times3$ input resolution. A single synthetic input tensor was generated per run using a deterministic seed (42) and held constant across all 2000 iterations. This methodology measures \textbf{steady-state warm-cache inference latency}---the best-case scenario for a deployed pipeline---rather than heterogeneous-input throughput, which would require a multi-image pool. We explicitly flag this scope limitation in Section~\ref{sec:discussion}.

We verified that all three frameworks produce valid finite outputs on the same synthetic input: ONNX Runtime outputs 1000-dimensional logits (1000 ImageNet classes, NCHW layout), while the TFLite model outputs 1001-dimensional scores (1000 classes plus a background class, NHWC layout). All outputs are NaN/Inf-free. Input preprocessing differs between frameworks (ONNX: $[-1,1]$ NCHW; TFLite: $[0,1]$ NHWC), which is standard for each framework's model distribution format. Since this benchmark measures inference \emph{latency} rather than prediction \emph{accuracy}, the exact output values do not confound the timing results.

\subsection{Benchmark Protocol}

Each framework was evaluated under single-threaded (1t) and multi-threaded (4t) configurations. The protocol for each run:

\begin{enumerate}
    \item \textbf{Warmup:} 1000 identical-iteration passes to stabilize CPU frequency scaling, branch predictors, and framework-internal caches.
    \item \textbf{Measurement:} 1000 identical passes with concurrent system-metric sampling.
\end{enumerate}

The Rust benchmark was compiled with \texttt{cargo build --release}. Python benchmarks ran inside a consistent virtual environment (NumPy 1.\textit{x}, SciPy 1.\textit{x}, psutil 5.\textit{x}). All 6 configurations were executed sequentially on an isolated Pi 5 to prevent CPU contention across concurrent processes.

\subsection{Data Collection and Statistical Analysis}

Per-iteration latency was recorded via high-resolution timers. System metrics were sampled every 200\,ms:
\begin{itemize}
    \item \textbf{CPU:} /proc/stat aggregated system-wide, reported as mean utilization across the measurement window.
    \item \textbf{Memory:} VmRSS from /proc/self/status.
    \item \textbf{Temperature:} /sys/class/thermal/thermal\_zone0/temp.
\end{itemize}

Statistical treatment:
\begin{enumerate}
    \item \textbf{Outlier removal:} Iterations outside $Q1 - 1.5 \times IQR$ or $Q3 + 1.5 \times IQR$ were removed.
    \item \textbf{Bootstrap CI:} 10\,000 resamples to estimate the 95\% confidence interval of the mean.
    \item \textbf{Summary:} Mean, median, standard deviation, min, max, throughput (fps), peak memory, and mean temperature.
\end{enumerate}

TensorFlow Lite achieves the lowest single-threaded latency at 22.19 ms---4.6$\times$ faster than Burn (tract-onnx) at 101.4 ms. ONNX Runtime occupies the middle ground at 45.35 ms. The narrow confidence intervals (width $<0.05$ ms for TFLite and ORT, $\sim0.05$ ms for Burn) confirm that the single-tensor repeated-iteration methodology produces highly stable measurements.

CPU utilization data (Table~\ref{tab:results}) provides the mechanistic explanation for the latency differences. Burn's tract-onnx uses only 23.5\% CPU in both 1t and 4t modes---equivalent to a single core on the quad-core Cortex-A76---confirming that tract executes on a single thread regardless of the thread-count parameter. TFLite and ONNX Runtime both show $\sim25\%$ single-threaded utilization rising to $\sim95\%$-$100\%$ with 4 threads, confirming effective parallelization.

Throughput mirrors latency: TFLite 4t achieves 93.3 fps, ORT 4t reaches 49.0 fps, while Burn remains at $\sim9.9$ fps due to its single-threaded execution model.

\subsection{Memory Efficiency}

Burn demonstrates exceptional memory efficiency at 41.8 MB peak RSS (Figure~\ref{fig:memory})---4.3$\times$ less than ORT (100 MB) and 12.9$\times$ less than TFLite (540 MB). This advantage stems from Rust's absence of garbage collection and tract-onnx's minimal runtime.

The TFLite memory footprint (540 MB, identical for 1t and 4t) warrants clarification. This value reflects the total process RSS including the Python interpreter, TensorFlow runtime library, and XNNPACK delegate initialization---not the model alone. The byte-identical values across thread configurations confirm that the primary memory cost is the Python environment and delegate allocation, which is incurred once at startup regardless of thread count. For deployment scenarios, the model-only memory footprint is approximately 14 MB for MobileNetV2 TFLite~\cite{sandler2018mobilenetv2}, with the remaining $\sim$525 MB being Python/runtime overhead.

\subsection{Thermal Characteristics}

All frameworks maintained temperatures 26--28$^\circ$C below the Pi 5's 85$^\circ$C throttle threshold. Multi-threaded runs showed a 6--8$^\circ$C increase over single-threaded baselines. Burn exhibited lower peak temperatures (51.8$^\circ$C) consistent with its lower CPU utilization, while TFLite 4t (58.4$^\circ$C) and ORT 4t (58.4$^\circ$C) ran warmer. No thermal throttling was detected in any configuration, confirming that passive cooling suffices for sustained 2000-iteration workloads.

\subsection{Multi-thread Scaling}

Figure~\ref{fig:speedup} quantifies the speedup from single-threaded to quad-threaded execution. TFLite achieves $2.08\times$ speedup and ORT $2.23\times$, approaching the theoretical $2.3\times$ Amdahl limit for the Cortex-A76's 4-issue superscalar pipeline under memory-bandwidth contention.

Burn shows a statistically significant \emph{regression} from 101.40 ms (1t) to 101.76 ms (4t). The 95\% CIs {[}101.37, 101.42{]} and {[}101.74, 101.79{]} do not overlap---$p < 0.05$---indicating a real albeit tiny multithreading overhead of 0.36 ms ($<0.4\%$). This is consistent with tract-onnx's single-threaded execution plan: setting \texttt{RAYON\_NUM\_THREADS=4} adds synchronization overhead without any parallel speedup.

\subsection{Threats to Validity and Scope Limitations}
\label{sec:discussion}

Several factors bound the generalizability of these results:

\begin{itemize}
    \item \textbf{Steady-state measurement:} The single-input repeated-iteration protocol measures warm-cache, branch-predictor-trained steady-state latency. Real deployments processing diverse inputs will experience additional latency from cache misses, memory allocation, and dynamic control flow. Our results should be interpreted as the \emph{baseline best case} for sustained inference.
    \item \textbf{Hardware generation:} The Raspberry Pi 5 (Cortex-A76) differs significantly from the Pi 4 (Cortex-A72); results on older ARM cores may change.
    \item \textbf{Model specificity:} MobileNetV2 represents efficient CNN architectures; transformer-based or recurrent models may show different scaling.
    \item \textbf{Framework version:} All frameworks are under active development. Performance characteristics may shift with future releases.
    \item \textbf{Output consistency:} The ONNX model (used by Burn and ORT) outputs 1000-dimensional NCHW logits, while the TFLite model outputs 1001-dimensional NHWC scores (extra background class). We verified all frameworks produce finite, non-NaN outputs but did not enforce bit-exact equivalence, as this benchmark measures timing, not accuracy.
\end{itemize}

\section{Conclusion}
\label{sec:conclusion}
%----------------------------------------------------------------------

This paper presented a systematic steady-state warm-cache benchmark of Burn (tract-onnx), TensorFlow Lite, and ONNX Runtime on Raspberry Pi 5. Key findings:

\begin{enumerate}
    \item TensorFlow Lite achieves the highest throughput (93.3 fps, 10.7 ms quad-threaded), optimal for latency-critical edge applications with sufficient memory budget.
    \item Burn (tract-onnx) demonstrates exceptional memory efficiency (41.8 MB RSS), a viable choice for memory-constrained deployments where maximum throughput is not the primary concern.
    \item ONNX Runtime provides a balanced middle ground (20.3 ms, 100 MB) with effective multi-thread scaling ($2.23\times$ speedup).
    \item All frameworks maintain stable thermal profiles under sustained 2000-iteration workloads with passive cooling.
    \item CPU utilization data confirms tract-onnx's single-threaded execution model (23.5\% on a quad-core CPU), explaining its lack of multi-thread scaling and providing a clear optimization target.
\end{enumerate}

The complete benchmark suite---including code, raw results including per-iteration latency arrays, figure-generation scripts, and the paper source---is publicly available at \url{https://github.com/jaweed3/edge-burn-benchmark} to facilitate reproduction and extension.

Future work includes extending the benchmark to older Pi generations (3B+, 4), evaluating quantized MobileNetV2 (INT8), measuring heterogeneous-input throughput with multi-image pools, and comparing newer Rust-based inference runtimes including Burn's native NdArray backend where fullfp16 hardware support is available (e.g., Apple M-series CPUs).
